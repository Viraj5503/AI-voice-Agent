## 1. STT fragment-as-turn segmentation — fixed without latency cost

**The bug surfaced in live console runs.**  A continuous caller statement
like *"I was driving on the A4 near Köln-Ost, and there was heavy rain,
and the other car hit me"* was getting emitted by Gradium STT as 3–4
separate fragments separated by short mid-sentence breaths.  livekit-
agents' default VAD-mode endpointing fired a turn-completion on the
first fragment, kicked off preemptive LLM generation, then had to
discard and retry as each subsequent fragment arrived — surfacing as
*"preemptive generation enabled but chat context or tools have changed"*
warnings in the logs and as Jamie answering a partial sentence first,
then "catching up" several seconds later.

**Three layers each addressing a different cause:**

1. **Semantic end-of-turn detection** — added `livekit-plugins-turn-
   detector` 1.5.6 and wired `MultilingualModel()` into `AgentSession.
   turn_detection`.  A small ~50ms ML model reads chat context plus
   the current transcript and predicts whether the caller is actually
   done speaking, regardless of pause length.  Wrapped in try/except
   so a missing plugin or non-job-context call site cleanly falls
   back to VAD-only endpointing.
2. **Dynamic endpointing** — switched from the deprecated
   `min_endpointing_delay`/`max_endpointing_delay` kwargs to
   `AgentSession(turn_handling={"endpointing": {"mode": "dynamic",
   "min_delay": 0.8, "max_delay": 4.0}})`.  Dynamic mode makes the
   timer self-tune to the caller's actual pause cadence.  `min_delay`
   widened from livekit's 0.5s default to 0.8s as the timing-only
   fallback when the semantic model is uncertain.
3. **Gradium STT tightened at the source** — `vad_threshold` from 0.6
   → 0.85, `buffer_size_seconds` from 0.08 → 0.12.  Gradium now needs
   more confident silence before declaring end-of-utterance, so mid-
   sentence breaths get batched into one transcript instead of
   fragmenting.

After a later main-branch rewrite to a custom `GeminiBrain` streaming
architecture (see § 4), the turn-detector hook stopped applying — the
new arch dispatches turns from a `user_input_transcribed` callback
rather than via `AgentSession.turn_detection`.  The two Gradium STT
knobs survive that pivot and were re-applied; the semantic model was
intentionally not re-added because it'd have no integration point.

**Result:** all three knobs env-overridable (`GRADIUM_VAD_THRESHOLD`,
`GRADIUM_BUFFER_S`, plus `ENDPOINT_MODE`/`MIN_ENDPOINT_DELAY_S`/
`MAX_ENDPOINT_DELAY_S` when the AgentSession-level hook is in use).

## 2. Pioneer / Fastino bounty — synthetic data + fine-tune evidence

**Decision: build the full synthetic-data → fine-tune → benchmark
pipeline with measurable F1 evidence.**

The competing extraction approaches all have trade-offs the bounty
rewards us for naming explicitly:

| Approach                       | Latency | Cost/call | F1   | On-device |
|---|---|---|---|---|
| Gemini Flash structured-output | 4400ms  | $0.0015   | 0.83 | ❌        |
| GPT-4o JSON mode               | 1200ms  | $0.0180   | 0.93 | ❌        |
| Regex / spaCy patterns         | <5ms    | $0.000    | low  | ✓        |
| **GLiNER zero-shot**           | 837ms   | $0.000    | 0.32 | ✓        |
| **GLiNER fine-tuned (jamie)**  | 116ms   | $0.000    | 0.48 | ✓        |

The fine-tuned 153M model (`urchade/gliner_small-v2.1` base) gets a
+50% F1 lift over the 530M zero-shot baseline AND runs 7.2× faster on
the same hardware.

**Notable trade-offs encountered along the way:**

- **Initial base was bi-large 530M** — measured 130s per training step
  on M-series MPS for 16 steps = ~35 minutes.  Too slow for a
  hackathon iteration loop.  Switched default to `urchade/gliner_small-
  v2.1` (153M); training dropped to ~10s/step.  The smaller base has a
  slightly lower zero-shot baseline, which the fine-tune more than
  recovers.
- **MPS OOM at step 10/16** — `RuntimeError: MPS backend out of memory
  (allocated 4.09 GiB, max 9.07 GiB)`.  Default flipped from `auto` to
  `cpu` for predictability on hackathon hardware.  CPU runs the small
  model in ~18 seconds for the same 16 steps.
- **`accelerate>=1.1.0` required by transformers.Trainer** — surfaced
  by an `ImportError` before any work was done.  Pinned in
  `requirements.txt`.
- **GLiNER saves config as `gliner_config.json`, not `config.json`** —
  the standard transformers naming.  `bench_gliner_finetuned()` was
  silently dropping the fine-tuned row from output until the existence
  check accepted either filename.

**Synthetic data generator design choice:** ask the LLM for transcripts
with **inline `[[label:phrase]]` markers**, then strip them and compute
word indices ourselves.  LLMs are bad at counting word indices; they're
good at producing natural text with structured spans.  Multi-provider
chain (Gemini direct → OpenAI-compat LLM_* → LLM_FALLBACK_*) auto-falls
through 429 / 5xx / 404, with model rotation for Gemini
(gemini-flash-latest → 2.5-flash → 2.5-flash-lite).

## 3. Gradium bounty — five non-trivial API surfaces, including one undocumented

The Gradium 900k-credits bounty rewards "best use of Gradium," and the
floor competition will be "we plugged in the LiveKit plugin."  The
project goes meaningfully past that.

**Three integration tiers, all live:**

1. **Gradbot** — the H0–H4 zero-config prototype path
   (`voice/gradbot_quickstart.py`).
2. **Direct SDK** — `voice/multiplex_demo.py` drives multiple concurrent
   TTS streams over a single WebSocket using `client_req_id` and
   `close_ws_on_eos=False` — the production property real call centres
   care about.
3. **livekit-plugins-gradium** — the production stack in
   `voice/livekit_agent.py`.

**Voice cloning** — `scripts/clone_voice.py` wraps `client.voice_create`
so a 10-second sample becomes a fresh `voice_id` with one command.

**Pronunciation dictionary via undocumented API** — Gradium has an
`/api/pronunciations/` endpoint that **isn't exposed in the SDK** and
isn't in the public docs WebFetch could see.  Discovered by probing
the API directly:

```http
POST https://eu.api.gradium.ai/api/pronunciations/
{
  "name": "jamie-fnol",
  "language": "en",
  "rules": [
    {"original": "FNOL", "rewrite": "eff-noll"},
    {"original": "DSGVO", "rewrite": "Day-Es-Gay-Fau-Oh"},
    {"original": "Vollkasko", "rewrite": "Foll-kass-ko"},
    ... 26 rules total
  ]
}
```

Returned uid passes to `gradium.TTS(pronunciation_id=uid)`.  Real
insurance pros pattern-match how Jamie says these acronyms within
seconds — getting "FNOL" wrong is a free "AI" vote.

**Tone tuning + STT noise fix** — `gradium.TTS(json_config={...})`
accepts `temp=0.85`, `padding_bonus=0.3`, `cfg_coef=2.2`,
`rewrite_rules="en"`.  `gradium.STT(temperature=0.0)` fixed a phantom-
transcript issue surfaced live (Whisper-style models hallucinate
"Marama", "I live in Chicago" from background ambient sound).

## 4. Architecture pivot: livekit-plugins-google → custom `GeminiBrain`

Mid-build the voice agent was rewritten to drop `livekit-plugins-
google`'s native LLM in favour of streaming through a custom
`GeminiBrain` directly, with a `user_input_transcribed` listener
instead of `on_user_turn_completed`.

**Why:**

- **Per-turn injection of CRM context, tool results, and claim state.**
  The native plugin creates the LLM call internally; the custom path
  lets us regenerate the system prompt every turn with the freshest
  state (filled pillars, tavily lookup results, last-Jamie-line anchor).
- **No silent provider drift.**  `agent.brain.make_brain()` is now
  Gemini-locked by default; cross-provider fallback to ollama/openai
  requires an explicit `BRAIN_ALLOW_NON_GEMINI_FALLBACK=1`.  This
  prevents the tuning-vs-evaluation mismatch where one part of the
  stack runs on Gemini and another on Groq, producing inconsistent
  outputs.
- **Heuristic Tavily fire path** stays available for small-model
  scenarios where LLM-driven function calling is unreliable.

**Trade-offs accepted:**

- The semantic turn-detector model from §1 stops applying.  Mitigated
  by re-applying the two surviving Gradium STT knobs.
- The provider-pluggable LLM (Gemini → Groq → Cerebras) is no longer
  active on the voice path; only Gemini.  Mitigated by making the
  stub branch return useful output rather than an empty string when
  the live key is rate-limited.

## 6. Tavily — 5-surface real-time grounding

`tools/tavily_lookup.py` exposes:

- `lookup_weather` — `topic="news", time_range="day", country="germany",
  include_answer="advanced"`.  The "magical" Turing-test moment.

**full country names**, not ISO codes — surfaced via an `Invalid country`
error in a live probe.  Switched to `country="germany"` everywhere.

## 7. Demo-day resilience patterns

- **LLM FallbackAdapter** for the voice path (Gemini primary, Groq
  fallback).  Different providers fail in different windows: Gemini
  hits per-minute quota first and recovers fast; Groq hits per-day
  tokens and recovers at midnight UTC; Anthropic depletes credit;
  Ollama OOMs.  The union is dramatically more reliable than any one.
- **Filler injection on listening→thinking** transition.  Real human
  agents say "mm-hmm" / "right" / "okay" the moment a caller stops
  talking; robots leave dead silence while the LLM produces its first
  token.  60% probability gate (`FILLER_RATE`) so it doesn't feel
  performative.
- **Provider-pluggable juror harness** (`tests/juror_bot.py`).  Tries
  Anthropic → OpenAI-compat → Gemini → stub.  Now Gemini-direct since
  Anthropic credit was depleted; uses a deliberately different model
  than Jamie's brain to avoid self-grading bias.
- **`load_dotenv(override=True)` everywhere** — empty shell exports
  were silently shadowing real values from `.env`.
